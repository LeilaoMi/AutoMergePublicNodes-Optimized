## 1. 总体判断

这个项目已经不是“简单订阅合并脚本”，而是一个完整流水线：

```text
29 个源 → 抓取 → 解析 → 去重 → GeoIP → 质量过滤 → TCP 预筛 → sing-box 真测 → 多格式输出 → 源审计 / 健康报告 → CI 自动提交
```

优点很明显：

1. **模块化已经成型**：`file core/fetcher.py`、`file parser.py`、`file tester.py`、`file generator.py`、`file geo.py` 分工清晰。
2. **测试意识强**：已有 19 个回归测试，覆盖多个曾经踩过的坑。
3. **输出面完整**：Clash、sing-box、V2Ray base64、URL 列表、converter 链接都有。
4. **真实可用导向**：用 sing-box 跑代理流量测试，而不是只测 TCP 握手。
5. **对 CI 现实有认知**：README 已明确说明 GitHub Actions 视角 ≠ 用户本地视角。

但现在项目的主要瓶颈也很清楚：

- **CI 自提交链路偏危险**：rebase 失败会自动 `force-with-lease`，还有 `force_push` 手动开关。
- **抓取超时降级有实锤 bug**：`fetch_all` 的 timeout 分支无法返回部分结果。
- **测试会污染仓库 tracked output**：运行 unittest 会修改 `file output/health_report.json`。

- **健康报告“报警但不失败”**：当前 low pass / real_test_errors 只是记录，不会挡住 CI。
- **GeoIP 依赖免费 HTTP ip-api**：可用但脆弱，且 HTTP 明文容易被中间链路篡改。
- **节点解析/生成的协议兼容边界还不够系统化**：靠回归测试兜几个点，但没有 corpus 化。

## P0-2. workflow 的 rebase 失败自动 force push 设计太危险

**位置**：`file .github/workflows/update.yml` 127-136 行附近。

**现状**

```bash
if git pull --rebase --autostash --strategy-option=theirs origin main; then
  git push origin HEAD:main && break
else
  git rebase --abort 2>/dev/null || true
  echo "::warning::rebase failed, falling back to force push"
  git push --force-with-lease origin HEAD:main && break
fi
```

还有手动输入：

```yaml
force_push:
  description: '强制推送（忽略冲突）'
  type: boolean
  default: false
```

**风险**

- 如果 main 同时有用户手动修复，CI 可能覆盖掉人的提交。
- `output/` 虽然是生成物，但 workflow 文件本身、README、配置也在同一分支上；force push 心智成本很高。
- `force_push` 暴露给手动运行，误点一次就可能造成历史混乱。

**建议**

优先级最高。改为：

1. 默认只普通 push。
2. rebase 冲突时失败并上传 artifact，不自动 force。
3. 如果真要 force，只允许维护者明确在 GitHub UI 手动输入字符串 `I_UNDERSTAND_FORCE_PUSH`，不是 boolean。
4. 更好：输出数据改走 `gh-pages` / `data` 分支，源码 main 不被 CI 反复提交。

## P0-3. `fetch_all` overall timeout 分支有实锤 bug，无法返回部分结果

**位置**：`file core/fetcher.py` 163-190 行。

**现状**

```python
tasks = [_wrapped(session, s) for s in sources if s.enabled]
try:
    return await asyncio.wait_for(asyncio.gather(*tasks), timeout=overall_timeout)
except asyncio.TimeoutError:
    done_results = [t.result() for t in tasks if t.done() and not t.cancelled()]
```

`tasks` 是 coroutine 列表，不是 `asyncio.Task` 对象，coroutine 没有 `.done()`。

**复现证据**

```text
fetch_all overall timeout (0.001s), returning partial results
AttributeError: 'coroutine' object has no attribute 'done'
```

**风险**

一旦整体抓取超时，设计上应该返回已完成源；实际会二次异常，整批失败。

**建议**

改为显式创建 task：

```python
tasks = [asyncio.create_task(_wrapped(session, s)) for s in sources if s.enabled]
try:
    return await asyncio.wait_for(asyncio.gather(*tasks), timeout=overall_timeout)
except asyncio.TimeoutError:
    done_results = [t.result() for t in tasks if t.done() and not t.cancelled() and t.exception() is None]
    for t in tasks:
        if not t.done():
            t.cancel()
    await asyncio.gather(*tasks, return_exceptions=True)
    return done_results
```

并加回归测试：强制 `overall_timeout=0.001`，断言不抛 AttributeError。

# P1 · CI / 发布链路安全

## P1-1. `contents: write` 权限作用域太宽

**现状**

```yaml
permissions:
  contents: write
```

**风险**

任何 workflow 里被第三方源内容影响的步骤，一旦出现注入链，具备写仓库权限。

**建议**

拆 job：

1. `validate/test/fetch` job：`contents: read`
2. `publish-output` job：`contents: write`，只在前面全部成功后运行。

---

## P1-2. 输出数据和源码共用 main 分支，冲突和回滚成本高

**现状**

CI 每 6 小时把 `output/*` commit 到 main。

**风险**

- main 历史巨大。
- 人改源码和 bot 改输出频繁交叉。
- 回滚源码时会带着 output 历史一起混乱。

**建议**

更合理结构：

```text
main      = 源码 / README / workflow
artifact  = output/* 自动产物
pages     = 可选 GitHub Pages 发布
```

Raw URL 可以切到 `artifact` 分支，README 写明。

## P1-3. `purge.jsdelivr.net` 失败被静默忽略

**现状**

```bash
curl -s "https://purge.jsdelivr.net/..." >/dev/null || true
```

**风险**

用户拿到旧缓存时，CI 看起来成功但实际订阅没更新。

**建议**

至少记录 HTTP 状态和失败文件：

```bash
code=$(curl -s -o /tmp/purge.json -w '%{http_code}' ...)
[ "$code" -lt 500 ] || echo "::warning::purge failed $f HTTP $code"
```

不要因为 purge 失败让整个 pipeline fail，但要能看见。

## P1-4. 没有保留上一次可用 verified 的“失败保护”

**现状**

`file health_report.py` 已经在 verified 0 时 exit 2，但如果某次输出质量极差，比如 verified 从 100 掉到 3，只要非 0 仍可能被提交。

**风险**

短暂网络异常会污染用户订阅。

**建议**

引入质量门槛：

```text
如果上次 verified >= 30，本次 < 上次 30%，则拒绝发布本次 verified，保留上一版。
```

可写入 `file stats.json`：

```json
"publish_guard": {
  "previous_verified": 80,
  "current_verified": 12,
  "action": "keep_previous"
}
```

## P1-7. workflow 手动输入参数没有上限保护

**现状**


Python 层也加上限。

## P2-3. `quality_prefilter` 注释和实际行为容易误解

**现状**

README 写：同 server+protocol 去重 + 同 IP 限 2。代码默认 `max_per_server=0`，但仍按 `(server, type)` 保留一个。

## P2-4. 下采样逻辑对协议不完全公平

**现状**

```python
per = max(args.test_limit // max(len(by_type), 1), 1)
for t, lst in by_type.items():
    sampled.extend(lst[:per])
nodes = sampled[:args.test_limit]
```

**问题**

- dict 顺序取决于输入顺序，源顺序会影响谁先进测试池。
- 如果某协议不足 `per`，剩余名额不会再分配给其他协议。
- 没有按 TCP 延迟、源质量、协议优先级综合排序。

**建议**

做二阶段采样：

1. 每协议按 TCP 延迟取基础配额。
2. 剩余名额放入全局池按 TCP 延迟/源通过率补齐。

# P3 · 性能 / 成本 / 仓库体积

## P3-1. output 大文件长期 commit 会让仓库快速膨胀

**当前规模**

- `file output/all.json` 十几万行
- `file output/all.yaml` 十万行级别
- 每 6 小时自动提交

**风险**

半年后仓库 clone 会越来越慢，Git 历史不可压缩。

**建议**

三选一：

1. output 改到单独 `artifact` 分支，定期 squash。
2. 使用 GitHub Releases / Pages artifact。
3. all 大文件只保留 `.txt/.urls`，json/yaml 只保留 verified。

## P3-2. `fetch_all` 20MB 单源上限仍偏高

**现状**

```python
MAX_FETCH_BYTES = 20 * 1024 * 1024
MAX_PARSE_BYTES = 10 * 1024 * 1024
```

**问题**

下载 20MB 后解析只用 10MB，浪费流量和内存。

**建议**

统一成：

```text
MAX_FETCH_BYTES = MAX_PARSE_BYTES = 10MB
```

或每源配置 `max_bytes`。

---

## P3-3. `tcp_check_batch` 一次性创建所有 task，节点多时内存压力大

**现状**

```python
results = await asyncio.gather(*[_check(n) for n in nodes], return_exceptions=True)
```

8095 节点会创建 8095 个 coroutine。

**建议**

改成队列 worker 模式，最多创建 `concurrency` 个 worker。

---

## P3-4. `SingBoxTester` 每个节点启动一个 sing-box 进程，成本高

**现状**

每个节点单独临时配置、单独进程。

**优点**：隔离好。

**缺点**：启动成本极高，CI 30 分钟限制下吞吐受限。

**中长期建议**

研究批量配置：一次 sing-box 配多个 outbound，每个 outbound 对应一个本地 mixed/socks 入站或 selector 切换。复杂度高，短期不建议硬改。

---

## P3-5. `_alloc_port` 先找再 bind 存在 TOCTOU 竞态

**现状**

先用 socket bind 找空闲端口，释放后再给 sing-box 使用。

**风险**

并发时端口可能被另一个进程抢占。概率不高，但存在。

**建议**

用全局 `asyncio.Lock` + 已分配集合，避免本进程内冲突；跨进程竞态只能接受或让 OS 直接分配端口但 sing-box 配置需要确定端口。

---

## P3-6. Geo DNS 解析串行按 chunk 批处理，但实际每 chunk 最多 50 域名

**现状**

```python
for i in range(0, min(len(domains), max_domain_resolves), 100):
```

默认 `max_domain_resolves=50`，只跑一轮。

**建议**

可以简化，直接：

```python
chunk = domains[:max_domain_resolves]
```

避免维护者误以为支持多批。

---

## P3-7. `write_outputs` 同时生成 all.json/all.yaml/all.urls/all.txt，all 对客户端并不都必要

**建议**

保留：

- verified 全格式
- all 只保留 `.urls` 和 `file .txt`
- all.json/all.yaml 可改为 release artifact 或手动生成

---

## P3-8. jsDelivr purge 串行执行，慢但不关键

**建议**

可用 `xargs -P 4` 并发 purge，并汇总失败。

---

## P3-9. stats 没记录各阶段耗时细分

**现状**

只有总 `duration_sec`。

**建议**

记录：

```json
"stage_durations": {
  "fetch": 12.3,
  "geo": 8.1,
  "tcp": 17.2,
  "real_test": 25.4,
  "generate": 1.0
}
```

这样才知道优化哪里最有效。

# P4 · 解析 / 生成兼容性

## P4-1. URL 参数没有统一 quote，部分特殊字符会生成坏订阅

**现状**

例如：

```python
params.append(f"sni={tls.get('server_name','')}")
params.append(f"host={host}")
```

如果 host/path/sni 包含特殊字符，可能破坏 URL。

**建议**

所有 query 参数用 `urllib.parse.urlencode`，不要手拼 `&`。

---

## P4-2. `node_to_url` 对 server 是 IPv6 时没有加方括号

**现状**

多处：

```python
{n.server}:{n.server_port}
```

**风险**

IPv6 会变成非法 URL。

**建议**

统一函数：

```python
def format_host(host):
    return f"[{host}]" if ":" in host and not host.startswith("[") else host
```

---

## P4-3. Clash / sing-box 生成配置没有二次语法验证

**现状**

health_report 只解析 JSON/YAML，不调用 sing-box/clash 检查配置可用性。

**建议**

CI 加：

```bash
./sing-box check -c output/verified.json
./sing-box check -c output/all.json   # 如果 all.json 继续保留
```

Clash Meta 如果没有二进制，可至少用 Mihomo check。

---

## P4-4. `node_to_clash` 的 hysteria/tuic/anytls 字段兼容性需跟随客户端版本测试

**风险**

这些协议字段在不同 Clash Meta / Mihomo / sing-box 版本中差异较大。

**建议**

建立 fixture corpus：每种协议准备 2-3 个真实样例，生成 YAML/JSON 后用目标客户端 check。

---

## P4-5. `parse_content` 对超大 base64 直接截断，可能截坏订阅

**现状**

```python
text = text[:MAX_PARSE_BYTES]
```

**风险**

base64 截断后可能整体不可解码，导致该源 0 节点。

**建议**

如果超过上限：

- 对纯文本：按行截断。
- 对 base64：先拒绝并记录 `too-large`，不要截断后误解析。

---

## P4-6. `parse_clash_proxy` 对非法端口 `int(...)` 可能抛异常，被外层 YAML 分支吞掉整份源

**现状**

```python
port = int(p.get("port", 0) or 0)
```

如果某个 proxy port 是奇怪字符串，`parse_clash_proxy` 抛异常。YAML 分支 `except Exception: pass`，可能整份 YAML 回退到 URL regex。

**建议**

单 proxy 粒度 try/except，不要让一条坏节点影响整源。

---

## P4-7. `url_re` 把 `https://` 当 HTTP 代理节点，可能误抓普通网页链接

**现状**

```python
(vmess|...|socks|https?)://...
```

**风险**

订阅源 README 或注释里的普通 `https://example.com` 可能被尝试解析成 http proxy，虽然通常没有端口会失败，但带端口的普通链接可能误入。

**建议**

HTTP/HTTPS 代理只在明确来源格式允许时启用，或要求含端口和认证/fragment。

---

## P4-8. 生成的 sing-box JSON inbound 仍含 `sniff`，和测试器配置策略不一致

**现状**

`tester.build_singbox_config` 已移除 inbound.sniff，避免 sing-box 1.13+ 问题。

但 `generator.write_outputs` 生成 Karing/sing-box 配置时仍有：

```json
{"type": "mixed", ..., "sniff": true, "sniff_override_destination": false}
```

**风险**

如果目标 sing-box 版本也废弃 inbound sniff，用户导入可能失败。

**建议**

用同样兼容策略调整输出配置，并加 `sing-box check -c output/verified.json`。

---

## P4-9. tag 截断后可能破坏用户识别信息

**现状**

`MAX_TAG_LENGTH = 48`，截断为 `(…)`。

**建议**

tag 结构化：

```text
🇺🇸 [258ms] source-name #短hash
```

用 hash 保留唯一性，少依赖原始长 tag。

---

# P5 · 可观测性 / 报告质量

## P5-1. `file health_report.json` 的 `ok=true` 但 alerts 有内容，用户容易误判

**现状**

本次 scratch health：

```text
ok=true
low_pass_protocols=6
real_test_errors 有 5 类
```

**建议**

拆成三个等级：

```json
"status": "ok" | "warning" | "critical"
```

- ok：无报警
- warning：有低通过率但输出可用
- critical：verified 为空或输出缺文件

---

## P5-2. `file stats.json` 没写 commit SHA / workflow run id

**建议**

CI 注入：

```json
"git": {
  "sha": "...",
  "run_id": "...",
  "run_attempt": 1
}
```

方便定位某次输出来自哪次 Actions。

---

## P5-3. `file source_audit.json` 没有记录源内容变化 hash

**现状**

只记录 nodes/bytes/duration/error/consecutive_dead。

**建议**

加：

```json
"content_hash": "sha256:...",
"fingerprint_hash": "sha256:..."
```

判断源是否长期“活着但不更新”。

---

## P5-4. 没有 per-source 真测贡献榜

**已有**

`source_pass_rate` 已在 stats 中写。

**建议**

在 README 或 `file health_report.json` 增加 Top/Bottom：

```json
"source_quality_top": [...]
"source_quality_bottom": [...]
```

方便删源/加权。

---

## P5-5. 日志中缺少“样本漏斗”一行总览

**建议**

每次 CI 输出：

```text
14524 raw → 8095 dedup → 8095 quality → 270 tcp → 33 real → 33 published
```

比多行散落更直观。

---

## P5-6. 错误原因聚合过粗

**现状**

`real_test_errors` 只按冒号前聚合。

**建议**

同时保留细分 Top 10：

```json
"real_test_error_details": [
  {"error": "204:status=403", "count": 120},
  {"error": "204:TimeoutError", "count": 50}
]
```

---

## P5-7. 没有记录各输出文件 SHA256

**建议**

`file health_report.json` 里写：

```json
"files": {
  "verified.yaml": {"size": 12345, "sha256": "..."}
}
```

用户反馈“订阅没变”时可快速判断 CDN 缓存问题。

## P5-8. 没有把 README 的“当前状态”自动更新

**建议**

```text

源健康
raw/dedup/tcp/real/verified

下载链接
```

## P5-9. 输出转换链接依赖第三方服务，缺少健康说明

**现状**

`file converter.md` 给了多个转换器。

**建议**

给每个转换器加说明：第三方服务可能记录 URL、不可用时换另一个；推荐优先直接用项目生成的 yaml/json。

## 结论

这个项目现在的方向是对的：**不要盲目追求 all 节点数量，而是用严格真测筛出尽量可信的 verified**。真正需要优先补的是“发布安全”和“故障保护”：避免一次 CI 异常污染用户订阅，避免 force push 覆盖人工修改，避免测试污染产物。

短期最值得做的不是再加源，而是把当前 29 个源的质量评分、失败保护、输出验证做扎实。这样项目会从“能跑的聚合脚本”升级成“可长期自动运行的订阅发布系统”。
