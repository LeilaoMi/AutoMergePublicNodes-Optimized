## 二、我补充发现的问题

### N1. `all.*` 输出会被 `verified` 节点改名污染

**严重级别**：高
**文件**：`file main.py`
**位置**：`dedup_pool = list(nodes)` 与后续 `n.tag = ...`

`dedup_pool = list(nodes)` 只复制了列表，没有复制 `Node` 对象。后面真实测试选中的节点会被原地改名：

```python
n.tag = f"{prefix}[{lat:>5.1f}ms{jitter_str}] {n.tag[:30]}"
```

这些同一个对象仍在 `dedup_pool` 里，所以 `all.*` 本应是“全量未测节点”，实际会混入一部分带国旗/延迟的 verified tag。

**复现结果**：模拟后 `file all.yaml` 输出变成：

```text
['🇺🇸 [100.0ms] orig1', 'orig2']
```

**建议**：在保存 `dedup_pool` 时深拷贝节点，或不要原地修改 `Node.tag`，生成 verified 输出前创建新 Node。

---

### N2. `parse_content()` 一行多个 URL 时只解析第一个可匹配 URL

**严重级别**：中
**文件**：`file core/parser.py`
**位置**：`parse_content()` 末尾 URL 提取逻辑

当前逻辑先 `re.findall()` 得到 scheme 列表，再 `line.find(scheme + '://')`，并且成功解析后立刻 `break`。

```python
for scheme in m:
    idx = line.find(scheme + "://")
    ...
    if n:
        nodes.append(n)
        break  # 一行只取第一个
```

如果某些订阅源一行里有多个节点 URL，只会保留第一个。

**复现**：

```text
vless://id@example.com:443?type=tcp#one trojan://p@host.com:443#two
```

解析结果只有 1 个 `vless`，`trojan` 被丢弃。

**建议**：让正则直接捕获完整 URL，并遍历所有 match；不要用 `line.find()` 回找 scheme。

---

### N3. 质量过滤无条件丢弃 80 / 8080 端口，可能误杀有效节点

**严重级别**：中
**文件**：`file main.py`
**位置**：`_PORT_BLOCKLIST = {0, 80, 8080}`

80/8080 虽然常见于明文或低质量节点，但也可能是可用 HTTP/SOCKS/WS 节点或伪装端口。现在默认 `--quality-filter=True`，会在真实测试前直接丢弃这些节点，无法通过后续测速证明是否可用。

**建议**：

- 至少不要对 `vmess/vless/trojan + ws/tls` 一刀切；
- 或把 80/8080 作为降权排序，而不是硬删除；
- 或在 README 里明确这是“激进过滤”。

---

### N4. `node_to_clash()` 类型标注不准确

**严重级别**：低
**文件**：`file core/generator.py`

函数签名是：

```python
def node_to_clash(n: Node) -> Dict[str, Any]:
```

但末尾会 `return None`。实际类型应为 `Optional[Dict[str, Any]]` 或 `Dict[str, Any] | None`。这会影响静态检查，也容易误导调用方。

---

### N5. README / 输出文件名出现敏感词替换残留，可能导致链接与实际文件名不一致

**严重级别**：中
**文件**：`file README.md`、`file main.py`、`output/*`

当前仓库里大量出现 `verif[用户触发屏蔽词]`，例如 README 链接、workflow purge 列表、`file main.py` 输出 prefix、`file output/verif[用户触发屏蔽词].yaml`。如果 GitHub 仓库真实文件名原本是 `verified.*`，这里的替换会导致链接和产物名异常；如果当前文件名就是替换后的字符串，也会影响用户使用和外部订阅转换器识别。

**建议**：统一确认真实目标文件名前缀。若应为 `verified`，需要全仓统一恢复；若是 Zo 显示层替换导致，提交前要在 GitHub 端核对真实内容。

---

### N6. `parse_content()` 对坏参数静默丢节点，缺少失败统计

**严重级别**：低 / 中
**文件**：`file core/parser.py`

例如：

```text
hysteria://example.com:443?auth=a&upmbps=abc&downmbps=50#bad
```

`parse_hysteria()` 内部 `int('abc')` 抛异常，`parse_url()` 捕获后返回 `None`，最终节点静默消失。订阅聚合类项目可以接受容错，但最好至少统计 parse failure 数量，方便发现源格式变化。

---

### N7. `geo_flag_map()` 只处理 IP 字符串，不解析域名

**严重级别**：低 / 中
**文件**：`file core/geo.py`

`geo_flag_map()` 直接把 `n.server` 当作 IP 送给 ip-api。很多节点 server 是域名，ip-api 对域名未必符合预期；同时 `_is_anycast()` 也只对 IP 前缀有效。结果是部分域名节点没有国旗或国家不准。

**建议**：如果确实需要地区分组，先 DNS 解析并缓存；或者接受“仅 IP 节点标旗”，并在 README 说明限制。
