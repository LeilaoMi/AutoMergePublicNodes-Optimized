| Q4 | `write_outputs()` 调 git subprocess | ✅ 成立 | **✅ 成立** | 输出生成函数内 `subprocess.check_output(["git", "remote", "get-url", "origin"])`，违反单一职责。无 git/无 remote 时有 fallback，但每次调用 fork 子进程。 |

## 三、本报告新增发现（X1–X2）

### X1. jsDelivr CDN 缓存清除列表遗漏 converter.md 文件

**严重级别**：中
**文件**：`file .github/workflows/update.yml`
**位置**：`Purge jsDelivr cache` 步骤

workflow 的 CDN 缓存清除列表包含 9 个文件：

```markdown
verified.txt verified.yaml verified.json verified.urls
all.txt all.yaml all.json all.urls stats.json
```

但实际输出目录有 12 个文件，以下 3 个**未被清除**：

- `file verified.converter.md`
- `file all.converter.md`
- `file source_audit.json`

**影响**：用户通过 jsDelivr CDN 访问转换器链接时，可能拿到过期的 converter.md 文件（虽然 converter.md 内容基于仓库 URL 生成，变化频率低，但 source_audit.json 每次审计都会更新）。

**修复**：在 purge 循环中添加缺失的文件名。

---

### X2. `_PROTO_PRIORITY` 包含 `wireguard` 但无对应解析器

**严重级别**：低
**文件**：`file main.py`
**位置**：`_PROTO_PRIORITY` 字典

```python
_PROTO_PRIORITY = {
    "vless": 0, "hysteria2": 1, "tuic": 2, "trojan": 3,
    "vmess": 4, "anytls": 4, "wireguard": 4,  # ← wireguard 有优先级
    ...
}
```

`wireguard` 在协议优先级字典中有定义（优先级 4），但 `URL_PARSERS` 和 `parse_clash_proxy()` 均无 wireguard 实现。这意味着 wireguard 节点永远不会进入管道，但优先级逻辑暗示开发者曾计划支持。

**影响**：无功能影响（wireguard 节点不会被创建），但代码意图不一致，属于 D1 的延伸。
