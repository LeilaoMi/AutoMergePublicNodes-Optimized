## 三、对 B2 严重性的复核（独立分析）

GLM 把 B2 列为"最需优先修复"，GPT 5.5 和 qwen3.7max 也都通过复现确认。但**经过我对 main.py 实际调用顺序的追溯，B2 的实际严重性应从 P0 降级为 P1**。

**main.py 实际执行顺序**:

1. `nodes = dedupe(all_nodes)`
2. `dedup_pool = list(nodes)` — 浅拷贝
3. ... GeoIP 查询 ...
4. ... 质量过滤、TCP、真实测试 ...
5. `for i, (n, lat, jit) in enumerate(top, 1): n.tag = f"{prefix}[{lat}ms] {n.tag[:30]}"` — 原地改 tag
6. **在调用** `write_outputs` **之前**:

   ```python
   for n in final_nodes:
       n.tag = _clamp_tag(n.tag)
   ```
7. `write_outputs(final_nodes, ..., prefix="verified")` → 内部先 `_ensure_unique_tags` 再写
8. `write_outputs(all_valid, ..., prefix="all")` → 同上

**实测结果**（`_PROTO_PRIORITY` 排序后构造两个 A×48+X / A×48+Y 的 tag）:

- clamp 后 n1 = `'AAAA…'`(48 字符)
- clamp 后 n2 = `'AAAA…'`(48 字符) — **完全相同**
- `_ensure_unique_tags` 后: n1 = `'AAAA…'`, n2 = `'AAAA…#1'`(48 字符)
- 后续 `p["name"]` / `ob["tag"]` 二次 clamp 仍保持唯一

**结论**:

- B2 的 clamp+去重**实际**能保证输出不冲突，因为 `file main.py` 在 `write_outputs` 前已做了统一 clamp + `_ensure_unique_tags` 是基于 clamp 后的 tag
- 但**问题描述本身仍然成立**：`write_outputs` 内部的 `_clamp_tag(p["name"])` / `_clamp_tag(ob["tag"])` 是冗余的，如果有人改动 `file main.py` 把预 clamp 步骤去掉，就会真的冲突
- **建议修复方向**: 移除 `write_outputs` 内部对 name/tag 的二次 clamp（重复处理就是 bug 隐患）；或者让 `_clamp_tag` 末尾自动去重

## 四、本报告新增发现

### M1. `file main.py` 原地修改 `n.tag` 存在隐性副作用（高优先级）

**文件**: `file main.py` 第 \~145-152 行

**代码**:

```python
for i, (n, lat, jit) in enumerate(top, 1):
    flag = flag_for_server(n.server, flag_map)
    prefix = f"{flag} " if flag else ""
    if lat > 0:
        jitter_str = f" +/-{jit:.0f}ms" if jit > 0 else ""
        n.tag = f"{prefix}[{lat:>5.1f}ms{jitter_str}] {n.tag[:30]}"
    final_nodes.append(n)
```

**问题**:

- `n` 是 `dedupe()` 返回列表中的对象，后续又被 `dedup_pool = list(nodes)` 浅拷贝共享
- 改 `n.tag` 同时污染 `final_nodes` 和 `dedup_pool`（即 N1）
- 此外，下游任何代码如果按**原 tag 查 source / 查审计 / 做对照**，都会因 tag 被改而查不到
- 例如 `node_source_map: Dict[str, str]`（`fingerprint` → `source`）用 `fingerprint` 作 key 是安全的；但如果有人想用 `n.tag` 做日志关联就出错

**建议**:

```python
# 在改名时构造新 Node
new_node = Node(
    type=n.type, tag=new_tag, server=n.server, server_port=n.server_port, raw=n.raw
)
final_nodes.append(new_node)
```

或更彻底地在 `run()` 入口对 `nodes` 做 `copy.deepcopy`，把"原对象"与"展示对象"彻底分离。

---

### M2. `file core/tester.py` `MIN_LATENCY_MS = 30` 硬阈值（低-中优先级）

**文件**: `file core/tester.py` 第 27 行

**问题**:

```python
MIN_LATENCY_MS = 30  # 低于此值视为同机房假节点
...
if avg_latency < MIN_LATENCY_MS:
    result.error = f"latency-too-low:{avg_latency:.1f}ms"
```

- 30ms 的硬阈值在跨太平洋链路（北美到亚洲）可能合理，但对**同机房 / 同区域代理**（如 AWS Tokyo → Cloudflare、阿里云上海 → 阿里云香港）可能误杀
- 用户订阅本地机房节点做出口时，30ms 是合理延迟
- 当前没有任何 CLI 参数能关掉这个判断

**建议**:

1. 改 `MIN_LATENCY_MS = 0` 或加 `--min-latency` 参数
2. 在 README 说明"低延迟节点会被视为可疑"（如确实想保留此行为）
3. 或者改为软警告（不拒掉，只在 stats.json 标记 `suspect_low_latency: true`）
