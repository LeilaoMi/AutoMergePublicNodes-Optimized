## 🔴 Bug（功能性缺陷）

### B1. `node_to_url()` 缺少 `http` 类型处理

**文件**: `core/generator.py`  
**位置**: `node_to_url()` 函数末尾

`node_to_url()` 有 `vmess`/`vless`/`trojan`/`shadowsocks`/`shadowsocksr`/`hysteria2`/`hysteria`/`tuic`/`anytls`/`socks` 的处理分支，但**完全没有 `http` 的处理**。HTTP 代理节点会直接返回空字符串 `""`。

```python
# 当前代码：http 类型直接落到 return ""
if t == "socks":
    ...
    return f"socks5://..."
return ""   # ← http 类型到了这里就丢了
```

**影响**:
- `.urls` 和 `.txt` 输出中丢失所有 HTTP 代理节点
- `node_to_clash()` 有正确处理 HTTP，但 `node_to_url()` 没有——**两种输出格式不一致**
- 如果用户订阅中有 HTTP 节点，Clash 格式能看到但 V2Ray/base64 格式看不到

**修复**: 在 `socks` 分支后添加 `http` 分支：
```python
if t == "http":
    userinfo = ""
    if r.get("username"):
        userinfo = f"{quote(r['username'])}:{quote(r.get('password',''))}@"
    scheme = "https" if r.get("tls", {}).get("enabled") else "http"
    return f"{scheme}://{userinfo}{n.server}:{n.server_port}#{quote(n.tag)}"
```

### B2. `_clamp_tag` 在 `_ensure_unique_tags` 之后仍可能制造 tag 冲突

**文件**: `core/generator.py` + `main.py`  
**位置**: `write_outputs()` 和 `main.py` 第 6 步

执行顺序有问题：
1. `main.py` 先调 `_clamp_tag(n.tag)` 截断 tag
2. `write_outputs()` 里再调 `_ensure_unique_tags(nodes)` 做去重（基于已截断后的 tag）
3. 然后 YAML 输出时又对每个 `p["name"]` 再调 `_clamp_tag()`
4. JSON 输出时又对每个 `ob["tag"]` 再调 `_clamp_tag()`

**冲突场景**: 两个节点的 tag 前 48 字符相同但之后不同：
- `node1.tag = "A"*48 + "X"` → clamp 后 `"A"*47 + "…"`
- `node2.tag = "A"*48 + "Y"` → clamp 后 `"A"*47 + "…"`（完全相同！）

`_ensure_unique_tags` 认为它们不同（因为原始 tag 不同），但后续 clamp 会让它们变成一样——导致**输出文件中出现重复 tag**，Karing/Clash 会报错或行为异常。

**修复**: 在 `_clamp_tag` 之后、`_ensure_unique_tags` 之前，不要重复调用 `_clamp_tag`。或者让 `_clamp_tag` 本身检查冲突。

---

### B3. `write_outputs()` 会修改传入的 nodes 列表（副作用）

**文件**: `core/generator.py`  
**位置**: `write_outputs()` 第一行 `_ensure_unique_tags(nodes)`

`_ensure_unique_tags` 直接修改了传入列表中 `Node` 对象的 `.tag` 属性。`write_outputs` 是一个"生成输出"的函数，但**副作用式地修改了输入**。

如果同一个 node 对象出现在两个不同的列表中（或函数被调用两次用同一列表），第二次调用时 tag 已经被加了 `#1` 等后缀，导致输出异常。

**修复**: 在函数开头对 nodes 做浅拷贝，或让 `_ensure_unique_tags` 不修改原始对象。

### B4. SSR URL 使用 `urlsafe_b64encode`，标准 SSR 客户端期望标准 base64

**文件**: `core/generator.py`  
**位置**: `node_to_url()` 的 `shadowsocksr` 分支

```python
password_b64 = base64.urlsafe_b64encode(r.get("password", "").encode()).decode().rstrip("=")
# ...
full = f"{head}/?{'&'.join(params)}"
return "ssr://" + base64.urlsafe_b64encode(full.encode()).decode().rstrip("=")
```

标准 SSR URL 格式使用**标准 base64**（`+/=`字符），但代码用了 `urlsafe_b64encode`（`-_/`替代字符）。虽然对字母数字密码两者结果相同，但当密码/用户名包含会产生 `+` 或 `/` 的字节时（中文、特殊字符等），urlsafe 编码会将 `+` 变成 `-`、`/` 变成 `_`——标准 SSR 客户端解码时会得到错误内容。

**影响**: 含特殊字符的 SSR 节点导出后无法被正确解析。

## 🟡 设计缺陷

### D1. Wireguard 和 Naive 协议只在文档/README 中宣称支持，但未实现

**文件**: `core/parser.py`  
**位置**: 文件头部 docstring + `main.py` 文件头部

文档声称支持 `wireguard`, `naive` 协议，但：
- `URL_PARSERS` 中没有 `wireguard` 和 `naive` 的 key
- `parse_clash_proxy()` 中没有 `wireguard` 和 `naive` 的处理分支
- `node_to_url()` 中没有 `wireguard` 和 `naive` 的处理
- `node_to_clash()` 中也没有

**现状**: 遇到 wireguard/naive 节点会被静默忽略，不会报错也不会警告。用户以为支持，实际上丢了节点。

**建议**: 要么实现支持，要么从文档和 README 中删除这两种协议的宣传。

---

### D2. `_extract_region()` 仅依赖 tag 首字符的国旗 emoji，无法处理无国旗节点

**文件**: `core/generator.py`  
**位置**: `_extract_region()`

函数逻辑：取 tag 前 2 个字符，判断是否为国旗 emoji（Regional Indicator Symbol），如果是则映射到地区名，否则归入"其他"。

**问题**:
- 项目在 `main.py` 中已经做了 GeoIP 查询（`geo_flag_map`），得到了 `{server: "🇸🇬"}` 的映射
- 但 `_extract_region()` **完全忽略了 `flag_map`**，只靠 tag 文字里的 emoji
- 全量输出（`all.*`）的节点 tag 中没有国旗 emoji（没经过 main.py 的改名步骤），所以分组时会被全部归入"其他"
- 即使 `verified.*` 的节点有国旗，`_extract_region()` 仍然重新解析 emoji 而不是复用已知结果

**建议**: 让 `_extract_region()` 接收 `flag_map` 参数，对没有国旗 emoji 的节点做 server IP 查表。

---

### D3. GeoIP 查询对 ip-api.com 无保护机制

**文件**: `core/geo.py`  
**位置**: `_lookup_batch()`

- ip-api.com batch 端点的限制是**每分钟 15 个请求**（不是 45）
- 代码中批间间隔只 1.5 秒，如果有 8000+ unique IP → 80+ 批 → 最快 120 秒跑完，但前 15 批（22.5s）用完后就会被 429，后面的全部失败
- 429 响应被 `except Exception: pass` 静默吞掉，不打印警告
- 如果 IP 不在 flag_map 中（包括 Cloudflare anycast 和查询失败的），节点就**没有国旗**，影响分组

**建议**: 
1. 批间间隔改为 4+ 秒（60/15 = 4s/batch），或改用付费版 API
2. 对 429 做退避重试（exponential backoff）
3. 缓存 GeoIP 结果到文件，避免每次运行重新查询
4. 备选方案：使用本地 GeoIP 数据库（如 `geoip2` + MaxMind GeoLite2）

### D4. `parse_content()` 对 base64 订阅的识别不够健壮

**文件**: `core/parser.py`  
**位置**: `parse_content()` 函数

当前 base64 识别逻辑（`is_b64()` 函数）：
- 要求长度 ≥ 16
- 正则匹配 `[A-Za-z0-9+/=\-_\\s]+`
- 解码后检查是否包含 `://` 或 `proxies:`

**问题**:
- 有些订阅服务返回的 base64 内容包含换行（每行 76 字符），`is_b64()` 先 `re.sub(r"\s+", "", text)` 再判断——这部分 OK
- 但 `stripped = re.sub(r"\s+", "", text)` 对**整个订阅内容**（可能几十 MB）做正则替换，效率低
- 如果订阅内容恰好以 `#` 开头（注释），前 200 字符内找不到 `://`，就跳过 base64 解码，直接进入逐行解析——可能丢失全部内容
- 先判断 YAML 再判断 base64，如果订阅既是有效 YAML 又是 base64 编码（虽然罕见），会优先当 YAML 处理

## 🟠 代码质量问题

### Q1. `parser.py` 用了 `Optional` 但没有导入

**文件**: `core/parser.py`

函数签名大量使用 `Optional[Node]`，但 `from typing import Any, Dict, List` 没有 `Optional`。  
虽然 `from __future__ import annotations` 使类型注解延迟求值，运行时不会报错，但：
- mypy/pyright 静态检查会报错
- 如果有人去掉 `__future__` 导入，代码立刻崩溃

---

### Q2. `main.py` 用了 `Tuple` 但没有导入

**文件**: `main.py`  
**位置**: L84

```python
async def tcp_check_batch(nodes: List[Node], concurrency: int = 200, timeout: float = 3.0) -> List[Tuple[Node, float]]:
```

同样有 `from __future__ import annotations` 保护，但 `Tuple` 没有从 `typing` 导入。

### Q3. `parse_content()` 的 URL 正则可能产生误匹配

**文件**: `core/parser.py`  
**位置**: `parse_content()` 函数中的 `re.findall`

```python
m = re.findall(r"(vmess|vless|trojan|ssr|ss|hysteria2|hy2|hysteria|hy|tuic|anytls|socks5|socks|https?)://[^\s<>'\"]+", line)
```

**问题**:
1. `ss` 可以匹配 `class://...` 中的 `ss://`，产生假 URL（虽然后续 parse 会丢弃，但浪费计算）
2. 替代项的顺序问题：`hysteria2` 在 `hysteria` 前面是正确的，但如果有人写了 `hy2://...`，正则也能匹配到 `hy2`
3. `https?` 可以匹配 `http://` 和 `https://`，但一般网页 URL 中也有这些 scheme，可能产生大量误匹配

**建议**: 增加 word boundary 或更严格的前缀匹配。

---

### Q4. `write_outputs()` 调用 `subprocess` 获取 git remote — 不应出现在纯函数中

**文件**: `core/generator.py`  
**位置**: `write_outputs()` 末尾

```python
import subprocess
try:
    remote_url = subprocess.check_output(["git", "remote", "get-url", "origin"], text=True).strip()
```

一个输出生成函数里调 `git` 命令：
- 在 CI 环境可能没有 git remote
- 在 Docker 环境可能没有 git
- 违反单一职责原则
- 首次 import 时不会失败，但每次调用都 fork 子进程

**建议**: 将 repo_path 作为参数传入，或从环境变量/配置文件读取。

### Q5. `fetcher.py` 使用 `ssl=False` 禁用了 SSL 证书验证

**文件**: `core/fetcher.py`  
**位置**: `fetch_all()` 和 `geo.py`

```python
connector = aiohttp.TCPConnector(limit=concurrency * 2, ssl=False)
```

全局禁用 SSL 验证意味着：
- 中间人攻击可以注入恶意节点数据
- 无法确认订阅源的真实性
- `geo.py` 中同样有 `ssl=False`

**建议**: 只对已知证书问题的源单独禁用，而不是全局关闭。

## ⚙️ 配置/运维问题

### C1. GitHub Actions 版本号可能不存在

**文件**: `.github/workflows/update.yml`

```yaml
- uses: actions/checkout@v6
- uses: actions/setup-python@v6
```

截至 2026 年 6 月，`actions/checkout` 和 `actions/setup-python` 的最新主版本是 v4。`@v6` 可能是笔误或超前指定。如果 GitHub 还没发布 v6，工作流会在 checkout 步骤就失败，**导致整个定时任务停止运行**。

**修复**: 确认可用版本，回退到 `@v4`。

---

### C2. `sources.yaml` 中的 Unicode 转义未正确渲染

**文件**: `config/sources.yaml`  
**位置**: 注释行

```yaml
# ===== v3.1: 10 \u4e2a\u65b0\u589e\u9ad8\u8d28\u91cf\u6e90 (2026-05 \u8c03\u7814) =====
```

YAML 默认会解析 `\uXXXX` 转义，但在注释中这虽然不影响功能，显示为乱码。应该直接用中文：`10 个新增高质量源`。

---

### C3. Workflow 的 `git push` 可能因并发冲突失败

**文件**: `.github/workflows/update.yml`

每 6 小时运行一次的 cron 任务 + push 触发 + 手动触发，并发设置为 `cancel-in-progress: false`。如果两次运行同时发生（如 cron 触发的同时用户 push 了代码），第二个 workflow 的 `git push` 可能因远程分支已更新而失败。

虽然提供了 `force_push` 选项，但默认不启用。失败后不会重试。

**建议**: 添加 `git pull --rebase` + 重试逻辑，或者默认启用 `--force-with-lease`。
